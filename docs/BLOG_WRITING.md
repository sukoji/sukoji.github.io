# 블로그 글 작성 가이드

에이전트·본인 모두 이 문서를 참고해 `_posts/` 글의 톤과 포맷을 맞춥니다.

**템플릿:** [`_posts/_TEMPLATE.md`](../_posts/_TEMPLATE.md)  
**예시 글:** [`_posts/2026-07-10-gpt-5-6-first-impressions.md`](../_posts/2026-07-10-gpt-5-6-first-impressions.md)

---

## 1. 파일 규칙

```
_posts/YYYY-MM-DD-slug.md
```

### Front matter (필수)

```yaml
---
layout: post
title: "구체적인 제목"
date: 2026-07-10
tags: [주제, 하위주제]
summary: 목록·SNS에 노출되는 한 줄 요약
---
```

| 필드 | 규칙 |
|------|------|
| `title` | 검색 가능하게. "메모", "정리", "나온 날" 같은 빈 제목·뉴스 레이아웃 지양 |
| `summary` | 1문장, 80자 내외 |
| `tags` | 2~4개, 소문자·하이픈. **연결 그래프**(`/graph/`)가 이 태그로 글·프로젝트를 잇는다. 프로젝트 태그(agents, eval, RAG 등)와 겹치게 쓰면 그래프가 촘촘해짐 |

## 2. 톤 (AI 티 줄이기)

> **본문만이 아니라 사용자에게 보이는 모든 문자열에 같은 기준을 적용한다.**
> bio, 소개(about), 소제목, `summary`, 캡션, front matter 문구까지 전부. 짧다고 "필드" 취급하지 말 것. 여기가 제일 눈에 띈다. 각 문장을 반드시:
> 1. **소리내 읽어** 어색하면 고친다.
> 2. **주어·목적어 확인** — "실제로 되는지" "어디서 막히는지"처럼 목적어·주어가 빠져 붕 뜨는 문장 금지. "무엇이 재현되는지" 처럼 대상을 명시한다.
> 3. **약어·고유명사는 처음 나올 때 한 줄 설명** — "Persode" 만 적지 말고 "에피소딕 메모리 저널링 에이전트 Persode" 처럼.
> 4. **기존 문구를 고칠 땐 단어만 갈아끼우지 말고** 의미부터 다시 쓴다. 어색한 뼈대를 물려받지 않는다.

**초안 쓴 뒤** [humanizer skill](../.cursor/skills/humanizer/SKILL.md) 기준으로 한 번 더 다듬습니다. (개인 스킬: `~/.cursor/skills/humanizer/`)

**문체:** 기본 `~합니다`체. `~한다`체·반말 섞지 않기. 단, 아래 "하지 말 것"의 단조로운 반복은 피합니다.

### 하지 말 것

- "혁신적인", "획기적인", "놀라운", "완벽한" 같은 마케팅 수식어
- 모든 섹션을 똑같은 길이·구조로 맞추기
- 같은 `~합니다` 종결을 3문장 연속 붙이기 (중간에 명사형·질문형·연결어미로 리듬 주기)
- 다른 블로그/보도자료 문장 **복붙** (출처 링크 + 자기 말로 요약만)
- 불필요한 이모지, 과한 bullet 남발
- "결론적으로", "요약하자면"으로 매 섹션 끝내기
- **기술 용어 억지 한글 음차** — "툴콜", "프롬프트 캐싱"의 "캐싱"까지 다 한글로 옮기지 말 것. 굳어진 영어 용어는 영어 그대로: `tool calling`, `prompt caching`, `tokenizer`, `reasoning`, `eval`, `orchestration`, `RAG`. 조사만 한글로 붙임 ("tool을 호출").

### 할 것

- **짧은 문장** — 한 문장에 경력·프로젝트·태도를 몰아넣지 않기
- **1인칭 경험** — "돌려봤습니다", "안 되면 로그만 남깁니다", "sympo에 넣어보면"
- **불확실성** — "아직", "직접 써봐야", "~일 것 같습니다", "맞는지 모르겠습니다"
- **자연스러운 결합** — "~연구를 수행합니다"처럼 딱딱한 명사+합니다 결합 지양. "eval 쪽 일을 합니다", "repo를 돌립니다", "만들었습니다"처럼 실제 쓰는 말
- **금지 패턴** — "궁금하면 ~", "남길 건 ~", "~쪽을 오가며", "직접 확인하는 시간을 많이 씁니다"
- **전문어 풀어주기** — 처음 나오는 용어는 한 번 쉽게 설명. 필요하면 비유 하나. 예: "티어(성능·가격이 다른 등급)", "5.6이 출시 연식이고 Sol/Terra/Luna가 Pro·기본·Lite 라인쯤".
- **본인 프로젝트에 엮기** — 새 기술을 sympo/token-stack 같은 실제 작업에 어떻게 붙일지로 연결. 남 얘기 요약이 아니라 내 파이프라인 관점.
- 마무리에 **다음 액션** 하나 (실험, 로그, 읽을 논문)

## 2.5 사실 확인 (필수)

초안 쓰기 **전에** 검증합니다. 이게 퀄리티의 절반입니다.

- 모델명·가격·출시일·기능 같은 **사실은 1차 출처**(공식 블로그·문서)로 확인. 뉴스는 보조.
- **지식 컷오프 이후** 사건이면 반드시 web search로 실존·수치 확인. 그럴듯하다고 단정 금지 (환각 위험).
- 확인 안 되는 건 "확인 필요"로 남기거나 빼기. 추정을 사실처럼 쓰지 않기.
- 참고 링크에 **확인에 쓴 1차 출처**를 반드시 포함.

> 예: GPT-5.6 글은 Sol/Terra/Luna·가격·프로그래매틱 tool calling을 공식 페이지+MarkTechPost로 확인한 뒤 작성. 컷오프 이후라 검색 없이 썼으면 전부 환각이었을 것.

## 3. 본문 구조 (권장)

```
1. 리드 (2~3문장) — 사건·배경
2. callout-summary — 한 줄 요약
3. 본문 섹션 (h2) — 2~4개
4. (선택) 명령어 / 코드 블록
5. (선택) callout-warn — 주의·한계
6. 참고 링크
```

### 소제목(h2) 규칙

목차에 그대로 뜬다. **스캔되는 명사구**로 쓴다. 의견·문장은 본문에 두고 제목에 넣지 않는다. 동시에 "소개/배경" 같은 텅 빈 제목도 피한다 (구체적인 명사구가 정답).

| 나쁨 (문장·의견) | 좋음 (명사구) |
|------------------|---------------|
| ultra는 돈 구멍일 수도 | ultra 병렬 모드 |
| Work는 뒤로 미룹니다 | ChatGPT Work |
| tool calling을 코드로 넘긴다 | Programmatic tool calling |
| 마지막에 몰면 늦다 | 끝에 몰린 검증의 한계 |
| 소개 / 배경 / 기타 | (구체적 주제 명사구로) |

제품·기능·개념 이름은 그 자체가 좋은 제목이다. "왜 X인가" 같은 질문형도 가끔 리듬용으로 허용. 마지막 액션 섹션은 "다음: 실패 스텝 찾기"처럼 다음에 뭘 할지 명사구로.

## 4. 시각·정보 계층 (마크다운)

### 한 줄 요약 박스

```markdown
> **적어 둠**
> 핵심 결론 한 문장.
{: .callout-summary}
```

("한 줄 요약" 대신 **적어 둠**, **메모** 등 가벼운 라벨 사용)

### 주의 박스

```markdown
> **주의**
> 실패 모드, 비용, 제한.
{: .callout-warn}
```

### 표 — 스펙·비교·가격

```markdown
| 모델 | 용도 | 가격 |
|------|------|------|
| **Sol** | ... | ... |
```

### 강조

- `**굵게**` — 모델명, 결론, 숫자 강조
- `` `코드` `` — API명, 파일명, 플래그
- 일반 링크 — 공식 문서·본인 repo 우선

### 명령어 블록 — **복붙 가능하게**

```markdown
```bash
# 용도 한 줄 주석
curl ... -d '{ ... }'
```
```

- 실제 동작하는 형태로 작성
- `$` 프롬프트는 넣지 않음 (복붙 방해)
- 긴 JSON은 최소 예시만

### 이미지

```markdown
![설명](/assets/images/posts/2026-07-10-slug.png)
```

- 스크린샷·다이어그램은 `assets/images/posts/`에 저장
- alt 텍스트 필수
- repo README 이미지 hotlink 가능하나, 오래 쓸 그림은 로컬 복사 권장

### 커버 이미지 (미리보기 카드)

front matter에 `image:`를 넣으면 **홈 미리보기 카드**와 **본문 상단 히어로**에 뜹니다. 없으면 카드는 제목 얹은 그라디언트로 대체됩니다.

```yaml
image: /assets/images/posts/YYYY-MM-DD-slug.svg
```

- **16:9** 권장 (`viewBox="0 0 1600 900"`). 카드가 `object-fit: cover`라 이 비율이어야 안 잘림.
- 가벼운 SVG 선호. `assets/images/posts/`에 `날짜-slug` 이름으로 저장.
- 본문 맨 위에 같은 파일을 `![alt](경로)`로 한 번 넣으면 글에서도 히어로로 보임.
- **로고·상표 그대로 복제 금지.** 브랜드 스타일을 살린 자체 제작 마크나 라이선스 이미지 사용.

## 5. 글자 색·스타일 (사이트 CSS 기준)

| 요소 | 렌더링 |
|------|--------|
| 본문 | 회색 톤 (`--ink-soft`) |
| **굵은 글** | 거의 검정 (`--ink`) |
| 링크 | teal (`--theme`) |
| `코드` | 연한 회색 배경 + 녹색 계열 글자 |
| 코드 블록 | 어두운 배경, 밝은 글자 |
| callout-summary | 연한 녹색 배경 |
| callout-warn | 연한 주황 배경 |

색은 마크다운 클래스로만 제어. HTML `<span style>` 사용 금지.

## 7. 주제별 체크리스트 (에이전트용)

작성 후 확인:

- [ ] `summary`가 front matter에 있음
- [ ] 첫 문단만 읽어도 주제가 이해됨
- [ ] callout-summary 1개 이상
- [ ] 표 또는 코드 블록 중 하나 이상 (기술 글일 때)
- [ ] 다른 글에서 문장 베끼지 않았음
- [ ] 참고 링크에 1차 출처 포함
- [ ] humanizer 스킬 기준으로 AI 패턴(병렬 구조, 빈 섹션 제목, ~합니다 연속) 제거했음
- [ ] `bundle exec jekyll serve`로 로컬 미리보기 확인
